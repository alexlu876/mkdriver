# Strategic pivot: BC → multi-track BTR (2026-04-17)

## Decision

Skip behavioral cloning entirely. Jump directly to Beyond The Rainbow (BTR) training, on all 32 vanilla MKWii tracks simultaneously, with a **track-agnostic single policy**. BC remains a future augmentation possibility (see "Future BC path" below) but is no longer on the critical path.

## Why the pivot

Original plan relied on recording 20+ user-driven Luigi Circuit demos to pretrain a BC policy before Phase 4 RL. User does not have the time budget to produce that volume of demonstrations. Without BC, there's no reason to stay single-track first — the architectural choice (stateful LSTM) that motivated the phased rollout is independent of multi-track support, and starting multi-track early avoids a lateral-migration Phase 3 later.

## Four sub-decisions (all confirmed 2026-04-17)

| Question | Decision |
|---|---|
| Multi-track strategy | **Track-agnostic single policy.** No track-id conditioning. Rely on encoder generalization. Motivation: maximum generality for future track expansion (incl. potential Retro Rewind later). |
| Savestate recording | **User will create all 32 manually** per `docs/SAVESTATE_PROTOCOL.md`. One-time ~2-hour chore. Non-blocking for initial Luigi-only smoke tests. |
| Compute | **Vast.ai committed upfront.** Local M4 is for code iteration / small smoke tests only; real training runs on rented GPU (RTX 4090 or equivalent). |
| BTR code reuse | **Fork into `src/mkw_rl/rl/btr.py`** from `third_party/Wii-RL/BTR.py` and modify in place. Same for `DolphinEnv.py` → `src/mkw_rl/env/dolphin_env.py`. Submodule stays for reference, but we own the code we run. |

## Research-grade uncertainty — be honest about this

**Track-agnostic multi-track MKWii BTR is not a solved problem.** VIPTankz published a single-track (Luigi Circuit) result and nothing on generalization. No MKWii RL paper we're aware of trains a track-agnostic policy across all 32 vanilla tracks. Possible failure modes:

- **Overfitting to early-sampled tracks.** If sampler happens to see Luigi + Mushroom Gorge a lot early, policy may never converge on Rainbow Road.
- **Destructive interference.** 32 tracks may be too diverse; the encoder can't find a shared representation. Loss plateaus or oscillates.
- **Reward-shaping leak.** VIPTankz's reward (progress + position) works on Luigi Circuit with specific course geometry; on tracks with shortcuts, loops, or anti-gravity segments, the reward may not align with "drive well."

If any of these happen, fallback is adding a track-id embedding (cheap retrofit to the encoder input) or going to a per-cup curriculum. **Plan to diagnose at ~2M-5M environment steps into training, not at 200M.**

### New Phase 2 (was Phase 4) scope

See `docs/TRAINING_METHODOLOGY.md` for the concrete algorithmic decisions (variable-per-track checkpoints, LSTM on BTR, lenient reset, progress-weighted sampler, reward-component structure). Summary:

1. Fork `third_party/Wii-RL/DolphinEnv.py` → `src/mkw_rl/env/dolphin_env.py`. Gym-compliant, multi-env-capable, `reset(track_slug=...)`.
2. Fork `third_party/Wii-RL/DolphinScript.py` → `src/mkw_rl/env/dolphin_script.py`. Port PAL RAM reads verbatim.
3. Implement `src/mkw_rl/env/reward.py` with variable-per-track checkpoint count (scaled by WR time from `data/track_metadata.yaml`), off-road/wall penalties, finish + position bonuses. Lenient reset: 1s progress / 15s window, no edge-fall termination.
4. Fork `third_party/Wii-RL/BTR.py` → `src/mkw_rl/rl/btr.py`. **Add LSTM on top of the IMPALA encoder** (v2 methodology — VIPTankz's published BTR is v1, frame-stack-only; v2 added LSTM as "the biggest change"). Use R2D2-style burn-in sequences in PER.
5. `src/mkw_rl/rl/track_sampler.py` — progress-weighted sampler (not uniform, not time-based).
6. `data/track_metadata.yaml` — per-track WR times (no-glitch category, user-curated from speedrun.com).
7. `docs/VAST_AI_SETUP.md` runbook for provisioning a 4090, uploading code + savestates, launching training, syncing checkpoints back.
8. `configs/btr.yaml` — VIPTankz's hyperparameters as starting point, plus LSTM-specific settings (`lstm_hidden`, `burn_in_len`, `learning_seq_len`).
9. Training monitoring: per-track reward curves (early detection of stalled tracks), per-component reward signals, gradient norms, replay buffer statistics, progress-weighted sampler distribution.

### What stays dormant but preserved

`src/mkw_rl/dtm/` and `src/mkw_rl/bc/` — full BC pipeline, tests, configs. Not deleted. These become **future BC augmentation scaffolding**. The action encoding in particular is directly reusable for BTR's discrete action space (21 steering bins × button combos).

## Future BC path (optional, post-BTR)

Once a working multi-track BTR policy exists, BC can come back as an auxiliary signal:

- **Source of demos: TAS community**, not the user. `data/raw/tas/` structure already exists. MKWii TAS files are available for most tracks — no user play time required.
- **Use it as an auxiliary loss** during BTR training: cross-entropy between policy logits and TAS-demo actions, weighted low. This is the standard "BC-augmented RL" pattern (DQfD, SAC+BC, etc.).
- **Use it as a warm start**: run the existing `src/mkw_rl/bc/train.py` on TAS demos, load the BC checkpoint as the initial BTR encoder weights. Requires matching the architecture exactly (IMPALA encoder + BTR IQN heads share the encoder; the LSTM head in BC doesn't transfer to BTR and would be dropped).
- **Trigger**: try this only if (a) vanilla BTR fails to generalize or (b) we want to close the gap to TAS-quality driving.
